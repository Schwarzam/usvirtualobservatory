Glossary of Technical Terms Related to VAO Logins

Index: A-E F-J K-O P-U V-Z

A-E

authentication
the action or process that must be taken to prove that an entity (e.g. user) is who they claim to be. A user often authenticates by providing a password; a web site authenticates itself to web browsers by presenting a certificate. Authentication involves at least two parties: an entity that claims to have a particular identity and an entity that needs to prove the identity in order to allow an interaction between them. The former usually presents credentials of some kind, and the latter will apply some test on the credentials to determine if they are authentic.
authorization
the permission (or the granting of permission) to take some action. Authorization refers to what a user is allowed to do. It is different from authentication in that it is assumed that a user's identity has already be verified independently of establishing what the user is allowed to do.
CA
short for certificate authority
CA certificate
the self-signed certificate that represents a certificate authority (CA) and which is used to sign the certificates it issues. A CA will distribute its CA cert freely distribute to all entities that choose to trust the CA. These entities can use the CA cert test the authenticiy of the certs it issues. By using the public key in the CA cert, the entity can prove that the signature in a issued cert is authentic.
cert
a shorter synonym for certificate
certificate
a particular type of credential, usually an X.509 certificate, that represents an identity and which can be encoded and delivered via a file.
certificate authority (CA)
a type of identity provider that issues certificate. A CA represents the root of the model of trust in network security: trust in particular CAs are assumed. A CA, through its CA certificate, can prove the authenticity of the certificates it issues.
Claimed Identifier
in the OpenID standard, an OpenID Identifier URL that represents the identity of a user that is chosen and controlled by the user. The OpenID protocol provides a mechanism for discovering the OP end-point URL and the corresponding OP-Local Identifier for the user. This allows the user to use a Claimed Identifier to log into an OpenID-supporting portal.
credential
a general term for a set of data that proves that an entity (e.g. a user, a service, or some other agent) has a particular identity--that is, they are who/what they claim to be. It can be in the form of a username and password, a certificate, or some other document that can be or has been verified to be authentic.
end-entity certificate
a type of X.509 certificate that has been signed directly by root Certificate Authority (CA). This is as opposed to a proxy certificate which has been signed by an intermediary. An end-entity certificate is often long-lived--i.e. it is valid for, say, a few years; consequently when with its private key, it must be stored securely.
entity
an actor or agent--like a user or a service--that can interact with other actors.

F-J

federated authentication (federated logins)
a common mechanism for authentication that a group of portals agree to support for logging in users. Within the federation, all portals support a common set of user identities. This means that a user can use the same identifier/username and password at any of the cooperating portals.
identity
in general, an attribute of an entity that allows it to be uniquely identified. Often, an identity is represented a unique name or label such as a username or URI.
identity provider
an organization that assigns identities to users and provides services to authenticate the identities of the users. These services could include a login service or a certificate issuing service.
IDP
short for identity provider

K-O

OAuth
an open community standard protocol for delegating permission to access a network resource to another entity on the network. In other words, it is a standard mechanism that enables a user to allow one web site (or tool or other agent) to access her private data on another site. In the protocol, the web site containing the private data will, after authenticating the user, create a secret token that is delivered to the entity needing access. That entity can then present the token to the web site as proof that it has permission from the user to access the private data. The tokens are usually short-lived (i.e. valid for a limited time) and can be made valid for access only to specific resources, as opposed to all the assets belonging to the user. OAuth, being about authorization, is independent from authentication. The required authentication step in the OAuth protocol can be done via one almost any technique, including OpenID.
OP
an OpenID Provider
OP end-point URL
the base URL used to access an OP's OpenID authentication service.
OP Identifier
stands for OpenID Provider Identifier
OP-Local Identifier
in the OpenID standard, the URL that is assigned to a user as his/her identity by the OpenID provider. For the VAO OpenID services, this URL has the form http://sso.usvao.org/openid/id/username.
OpenID
an open community standard protocol for federated, single sign-on authentication. By being federated, a group of portals agree to support a single set of identities and a common way to authenticate users; this means that a user can use the same identifier/username and password at any of the cooperating portals. Single sign-on means that once the user logs into one portal in the federation and starts a session, the user does not need to re-type a password to connect to another portal in the federation.
OpenID identifier
a URL that represents the identity of a user that can be used to log into a portal that supports OpenID.
OpenID provider
an identity provider that supports an OpenID-compliant authentication service. An OP provides the web site where an OpenID user enters their password.
OpenID Provider Identifier
an OpenID identifier that represents an OpenID provider itself. The OpenID standard dictates that when a user (or agent acting on the user's behalf) attempts to authenticate with the OP Identifier (as opposed her personal identifier), the OP should present an interface that lets the user choose which identity to authenticate as. In the VAO federation, when a portal uses the VAO's OP Identifier, the user will be presented a form where they can enter their user name rather than their whole OpenID identifier.

P-U

portal
a collection of related documents and services available through a web site and accessed with a browser. In the context of VAO Logins, a portal is specifically such a collection that requires logging in (i.e. authenticating) in order to access it. Logging in is the action that allows a user to "enter" a portal. In OpenID-speak, a portal is an relying party (RP).
private key
as part of public-private key authentication, it is the token that is used to decrypt a message. The entity that claims to have some identity holds the private key and uses it to decrypt a "challenge" message by others wishing to authenticate their identity. Since successful decryption proves possession of the private key, it is the key to the identity; thus, it must be stored securely or others could steal the private key and assume the entity's identity.
public key
as part of public-private key authentication, it is the token that is used to encrypt a message. The entity that wishes to prove another's identity is given a public key that it will use to encrypt a message. It will send that message to the entity claiming the identity as a challenge: the entity that can decrypt it proves that they have the associated private key. Public keys are intended to passed around freely; however, as part of a certificate, the key is signed by a trusted entity (e.g. a CA). The signature essentially indicates that the holder of the private key has the identity described in the certificate.
public-private key authentication
the authentication mechanism at the core of X.509 certificate authentication. It uses a type of encryption that involves two keys: a "public" one for encrypting and and a "private" one for decrypting. In the authentication process the entity claiming an identity presents a certificate that contains the public key. The "challenging" entity wishing to prove the identity encrypts a message using the public key and gives it back to the claiming entity. The claiming entity has the private key and uses it to decrypt the message and send back the result. By demonstrating decryption, the claiming entity proves they have the private key.
proxy certificate
a type of X.509 certificate that is that is a temporary stand-in for a long-term certificate. A proxy certificate has been signed using an intermediate rather than by a root CA (as with an end-entity certificate). A proxy is usually presented as the end of a chain of certificates in which each has been used to sign the certificate after it; at the other end of the chain is a certificate signed by a trusted entity (e.g. a root CA).

A proxy certificate is created for actual service interactions to be safer than an end-entity certificate because it is shorter-lived--i.e. valid for, say, only a few hours.

RP
a relying party
relying party
a portal or other software entity that wishes to authenticate users through an OpenID identity provider to allow them to access protected resources.
SSL
stands for Secure Socket Layer.
Secure Socket Layer
essentially a synonym for Transport-Layer Security. SSL is the standard encryption protocol used between web sites and web browsers. Essentially, after a successful TLS "handshake" that establishes the secure connection, the subsequent HTTP messages are encrypted before being sent.
SSO
stands for single sign-on
single sign-on
refers to a feature of federated authentication such that when a user logs into one portal in the federation and starts a session, the user will not have to re-type his password when logging into another federated portal. Similarly, if the portal accesses another protected resource on the user's behalf, the user will not need to re-type her password to authenticate.
TLS
stands for Transport-Layer Security
Transport-Layer Security
a standard protocol for encrypting messages sent in socket communication. In TLS, the identity of one or both of the two parties through a special "handshake" where X.509 certificates are exchanged and validated; upon successful validation, all subsequent messages are encrypted to prevent spying or corruption by other parties.

V-Z

X.509 certificate
a type of certificate that takes a standardized form. It is the most common form of certificate in use in the internet. Information in such a cert includes an identifier for the entity it represents, the identifier for the issuer (the entity that assigned and validated the represented entity), a date and time until which the certificate is valid, and a public key used to create challenges to prove the presenting entity has the stated identity.
