Low-Level (HTTP) Interface

Many users are best served by direct programmatic access to the cross-comparison functionality. In fact, this project's own GUI needs nothing else. Here we describe in detail how that interface works and how to use it.

Interacting with the cross-comparison service is actually a three-step process:

  • Upload a table. This includes vetting the data: format conversion if necessessary and deciding whether the table has enough information to determine source coordinates. The upload is usually the longest single step in the whole cross-comparison process, though if the service has to determine coordinates via object names in the records, this can be very lengthy.
  • The Cross-comparison itself.
  • Data retrieval. The cross-comparison return is just a summary of the processing with links to results. The actual data retrieval includes translation of the data to the desired output format.

Each has a simple HTTP GET/POST interface with an XML return structure (except data retrieval, which just returns the data). There are also interfaces to retrieve an XML listing of the tables the user currently has in the upload space and an XML summary of the archived catalogs that can be compared against. This space is temporary (files older than four days get deleted) but it allows the user to reuse uploaded tables for multiple comparisons. The same rules apply to result data.
